# Codex Project Patch Prompt

Исполни этот патч в текущем репозитории полностью и без дополнительных вопросов, если только не обнаружишь реальный блокер.

## Цель

Сделай репозиторий полностью готовым к работе в Codex IDE/CLI на уровне проекта, не затрагивая глобальную конфигурацию пользователя.

Нужно:

1. Добавить project-local Codex bootstrap.
2. Перенести MCP из Claude-формата в Codex-формат.
3. Перенести repo-local skills из Claude-структуры в Codex-структуру.
4. Максимально перенести локальные Claude rules/permissions в Codex rules.
5. Добавить проверочный `check_up_codex.md`.
6. Добавить sync-скрипты для MCP, skills и rules.
7. В конце удалить **этот файл** из корня проекта.

## Жёсткие ограничения

- Не изменяй `~/.codex/config.toml`, глобальные настройки VS Code или глобальные user skills.
- Всё должно быть **только на уровне репозитория** (согласуется с секцией «Границы файловой системы и БД» в `CLAUDE.md`: чтение/запись только в папке текущего проекта, без установки пакетов и загрузки из интернета).
- Не ломай существующие пользовательские изменения.
- Если в репозитории уже есть Codex-конфигурация, **мерджи** её аккуратно, а не перезаписывай вслепую.
- Используй `.mcp.json` как источник истины для MCP.
- Если есть `AGENTS.md`, считай его основным project doc для Codex.
- Если `AGENTS.md` нет, используй `CLAUDE.md` через `project_doc_fallback_filenames` — это важно: в `CLAUDE.md` находится секция «Границы файловой системы и БД», которую Codex-агент должен видеть и соблюдать.
- Не трогай `.dev.env` и не выводи значение `DB_PWD` ни в логах, ни в отчёте о переносе.

## Что нужно исследовать перед правками

Перед изменениями проверь наличие и содержимое:

- `.mcp.json`
- `CLAUDE.md`
- `AGENTS.md` при наличии
- `.claude/settings.local.json`
- `.claude/skills/`
- `check_up.md`
- `.gitignore`
- `.codex/` и `.agents/` при наличии

Если часть файлов отсутствует, не останавливайся: делай best-effort реализацию и явно фиксируй принятые допущения в финальном отчёте.

## Обязательные изменения

### 1. Project-local Codex config

Создай или обнови `.codex/config.toml`.

Требования:

- добавь строку схемы:

```toml
#:schema https://developers.openai.com/codex/config-schema.json
```

- настрой project docs:
  - если `AGENTS.md` есть, не ломай его;
  - если `AGENTS.md` нет, добавь:

```toml
project_doc_fallback_filenames = ["CLAUDE.md"]
project_doc_max_bytes = 65536
```

- перенеси все MCP-серверы из `.mcp.json` в `[mcp_servers.<name>]` блоки Codex-конфига;
- для MCP используй только repo-local конфиг;
- не выставляй `required = true`, если в проекте явно нет такого требования;
- если в конфиге уже есть другие Codex-настройки, сохрани их.

### 2. Sync для MCP

Создай `scripts/sync-codex-mcp.ps1`.

Скрипт должен:

- читать `.mcp.json`;
- валидировать, что серверы можно отразить в Codex-конфиге;
- обновлять **только помеченный generated-блок** MCP внутри `.codex/config.toml`;
- сохранять schema line и project-doc настройки;
- быть идемпотентным;
- не трогать глобальные файлы.

### 3. Порт skills в Codex

Если есть `.claude/skills/`, создай repo-local Codex skills в `.agents/skills/`.

Стратегия переноса:

- по умолчанию **копируй** навыки из `.claude/skills/` в `.agents/skills/`, а не делай symlink, если репозиторий уже не использует symlink-стратегию;
- сохраняй `SKILL.md`, `scripts/`, `references/`, `assets/`, `agents/` и прочие полезные файлы;
- не вычищай frontmatter без необходимости: если skill валиден для Codex, оставь его максимально близким к оригиналу;
- если отдельный skill содержит Claude-specific поля или инструкции, которые могут мешать Codex, адаптируй минимально необходимым образом и зафиксируй это в отчёте;
- если `.agents/skills/` уже существует, аккуратно обнови/смёрджи её, не снося unrelated содержимое.

### 4. Sync для skills

Создай `scripts/sync-codex-skills.ps1`.

Скрипт должен:

- зеркалировать `.claude/skills/` в `.agents/skills/`;
- корректно создавать новые навыки;
- обновлять изменённые;
- удалять из `.agents/skills/` только те элементы, которые были сгенерированы из `.claude/skills/` и больше там не существуют;
- быть идемпотентным;
- по возможности не трогать вручную созданные Codex-only навыки.

Если для безопасной синхронизации нужен файл-маркер, manifest или generated-папка, создай его.

### 5. Порт rules / permissions из Claude в Codex

Создай repo-local Codex rules.

Предпочтительное расположение:

- `.codex/rules/project.rules`

Что переносить:

- анализируй `.claude/settings.local.json`, секцию `permissions.allow`;
- **дополнительно учитывай секцию «Границы файловой системы и БД» в `CLAUDE.md`** — она политика проекта, а не permissions, но Codex rules не должны ей противоречить. В частности: не создавай allow-правил, которые разрешили бы выход за пределы папки проекта, установку пакетов, загрузку из интернета или подключение к базам 1С, отличным от заданной в `.dev.env`;
- переводи в Codex rules только то, что имеет **прямой или безопасный аналог**;
- для узких командных префиксов создавай `prefix_rule(... decision = "allow")`;
- для слишком широких или рискованных правил создавай `decision = "prompt"`, а не `allow`;
- не делай опасно широких allow-правил наподобие безусловного разрешения для всего `python`, `bash`, `powershell`, если это не обосновано и не является узким эквивалентом;
- MCP tool names из Claude не пытайся механически переводить в Codex rules, а учитывай их через MCP config / skills / отчёт о переносе.

Если часть Claude permissions не имеет прямого аналога:

- всё равно создай **лучший безопасный Codex-аналог** для переносимой части;
- оставшуюся часть зафиксируй как `needs translation` или `no direct equivalent` в отдельном отчёте.

### 6. Sync / export для rules

Создай `scripts/sync-codex-rules.ps1`.

Скрипт должен:

- читать `.claude/settings.local.json`;
- генерировать или обновлять `.codex/rules/project.rules`;
- делать best-effort translation;
- явно помечать сгенерированный блок;
- быть идемпотентным.

Если для части правил нельзя сделать автоматический перенос без риска, сгенерируй их как `prompt` и добавь комментарий почему.

### 7. Отчёт о переносе

Создай файл:

- `meta/codex-port-report.md`

В нём зафиксируй:

- что было перенесено автоматически;
- какие skills были скопированы как есть;
- какие skills потребовали адаптацию;
- какие Claude permissions/rules переведены в Codex rules;
- что осталось без прямого аналога;
- какие ручные шаги ещё нужны пользователю.

Формат отчёта должен быть коротким, но конкретным.

### 8. Codex check-up prompt

Создай `check_up_codex.md`, ориентируясь на `check_up.md`, но адаптируй его под Codex.

Он должен проверять минимум:

- trusted project;
- `.codex/config.toml`;
- паритет `.mcp.json` vs `.codex/config.toml`;
- доступность MCP;
- `.claude/settings.local.json` vs `.codex/rules/project.rules`;
- `.claude/skills/` vs `.agents/skills/`;
- drift и следующие шаги.

### 9. .gitignore

Проверь `.gitignore`.

Если он блокирует нужные repo-local Codex файлы, исправь это так, чтобы можно было трекать:

- `.codex/config.toml`
- `.codex/rules/`
- `.agents/skills/`
- `check_up_codex.md`
- новые `scripts/sync-codex-*.ps1`
- `meta/codex-port-report.md`

Не добавляй лишних ignore-правил без необходимости.

## Проверки после реализации

После правок выполни проверки:

1. Убедись, что `.codex/config.toml` существует и содержит project-local MCP.
2. Запусти `scripts/sync-codex-mcp.ps1` минимум два раза и проверь идемпотентность.
3. Запусти `scripts/sync-codex-skills.ps1` минимум два раза и проверь идемпотентность.
4. Запусти `scripts/sync-codex-rules.ps1` минимум два раза и проверь идемпотентность.
5. Проверь, что `.agents/skills/` создана и содержит перенесённые навыки.
6. Проверь, что `.codex/rules/project.rules` создан.
7. Проверь, что `meta/codex-port-report.md` создан.
8. Не меняй unrelated файлы без причины.

Если какая-то проверка упирается в sandbox/permissions, не бросай задачу: безопасно эскалируй только то, что реально нужно для завершения.

## Итоговый формат ответа

В финале дай короткий отчёт:

- что создано/обновлено;
- что было перенесено по MCP;
- что было перенесено по skills;
- что было перенесено по rules;
- что не удалось перенести 1:1;
- какие проверки были выполнены;
- что нужно сделать пользователю вручную после патча.

## Обязательная финальная операция

После того как всё реализовано и проверено, **удали этот файл из корня проекта**.

Если имя файла отличается от ожидаемого, удали именно файл, из которого был взят этот prompt.
