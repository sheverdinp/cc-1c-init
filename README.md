# cc-1c-init — промт для инициализации 1С-проектов

Промт для Claude Code, который за одну команду разворачивает полную структуру 1С-проекта с AI-инструментарием: стандарты кода, ролевые агенты, навыки (skills), MCP-интеграция, скрипты обновления.

## Зачем

Настройка нового 1С-проекта под Claude Code — рутина: создать каталоги, написать CLAUDE.md, установить навыки, сконвертировать правила из Cursor-формата, настроить .gitignore, подготовить шаблоны. Этот репозиторий автоматизирует всё это одним промтом.

## Что создаёт промт

### Структура каталогов

```
src/
  config_base/       — snapshot основной конфигурации (read-only)
  config_ref/        — snapshot-ы внешних конфигураций (read-only)
  extensions/        — расширения
  reports/           — внешние отчёты
  processings/       — внешние обработки
tasks/
  active/            — текущие задачи (формат T-XXXX_name)
  archive/           — завершённые задачи
kb/
  1c-rules/          — стандарты, антипаттерны, MCP-справочник
  1c-rules/agents/   — 12 ролевых агентов (аналитик, архитектор, разработчик...)
sandbox/             — эксперименты и черновики
meta/                — контекст проекта, параметры конфигурации
scripts/             — доверенные скрипты (обновление навыков и правил)
tools/               — вспомогательные утилиты
tmp/                 — временные файлы
releases/            — релизы
.claude/skills/      — навыки Claude Code (cc-1c-skills)
```

### Ключевые файлы

- **CLAUDE.md** — правила проекта, протокол анализа кода, границы ФС/БД, ссылки на стандарты и агентов
- **.dev.env.example** — шаблон параметров проекта (префикс, компания, версия платформы, шаблоны комментариев, параметры подключения к БД `DB_SRVR`/`DB_REF`/`DB_USER`/`DB_PWD`)
- **.gitignore** — настроенный под 1С-разработку (бинарники, снапшоты, кэши, `.dev.env` с секретами)
- **check_up.md** — промт диагностики проекта (структура, обновления, качество кода)

### Границы работы AI-агента

`CLAUDE.md` фиксирует жёсткие границы, чтобы агент с широкими правами не вышел за пределы проекта:

- **Файловая система** — чтение и запись только в папке текущего проекта; запрет на установку пакетов и загрузку из интернета.
- **База 1С** — подключение только к базе из `.dev.env` (`DB_SRVR`/`DB_REF`/`DB_USER`/`DB_PWD`). Разрешены операции через навыки `.claude/skills/` и штатный цикл разработки расширений (`DumpConfigToFiles`, `LoadConfigFromFiles`, `UpdateDBCfg`, загрузка внешних отчётов/обработок, `CheckModules`). Подключение к другим базам, выполнение произвольного кода, массовые изменения данных и админ-операции — только с явного подтверждения в сессии. Работа в продуктивной базе запрещена.
- **Секреты** — `DB_PWD` нельзя логировать, выводить в отчётах или коммитить; `.dev.env` исключён из git.

### Внешние зависимости

Промт автоматически клонирует и устанавливает:

- **[cc-1c-skills](https://github.com/Nikolay-Shirokov/cc-1c-skills)** — навыки для полного цикла разработки 1С (слэш-команды: `/epf-init`, `/epf-build`, `/cfe-init` и др.)
- **[cursor_rules_1c](https://github.com/comol/cursor_rules_1c)** — стандарты кодирования, антипаттерны, агенты (конвертируются из формата Cursor в формат Claude Code)

## Как использовать

Рабочий процесс — **два промта поочерёдно**: сначала `promt_new_project.md` в Claude Code, затем `patch_codex.md` в Codex. Первый разворачивает структуру проекта и устанавливает инструментарий Claude Code, второй — портирует конфигурацию в Codex (project-local `.codex/config.toml`, MCP, skills, rules, check-up), не затрагивая глобальные настройки пользователя.

### Новый проект

1. Создай пустой каталог и инициализируй git:
   ```bash
   mkdir my-1c-project && cd my-1c-project
   git init
   ```

2. Скопируй `promt_new_project.md` и `patch_codex.md` в каталог проекта.

3. **Шаг 1 — Claude Code.** Запусти Claude Code и передай первый промт:
   ```bash
   claude
   ```
   ```
   Выполни инструкции из promt_new_project.md
   ```
   После завершения промт удаляет себя из корня проекта.

4. **Шаг 2 — Codex.** Открой тот же репозиторий в Codex IDE/CLI и передай второй промт:
   ```
   Выполни инструкции из patch_codex.md
   ```
   Патч создаст `.codex/config.toml`, `.agents/skills/`, `.codex/rules/project.rules`, sync-скрипты и `check_up_codex.md`, а затем удалит себя из корня проекта.

5. Заполни параметры проекта:
   - Скопируй `.dev.env.example` → `.dev.env` и укажи `PREFIX`, `COMPANY`, версию платформы, параметры подключения к БД (`DB_SRVR`, `DB_REF`, `DB_USER`, `DB_PWD`)
   - Заполни `meta/project_context.md` данными проекта

### Патчи для существующих проектов

Если проект уже инициализирован предыдущей версией промта, используй точечные патчи — они идемпотентны и не трогают существующие значения:

- `promt_patch_db_boundaries.md` — добавляет секцию «Границы файловой системы и БД» в `CLAUDE.md` и параметры подключения к БД в `.dev.env` / `.dev.env.example`.
- `patch_codex.md` — портирует уже существующий Claude-setup в Codex (MCP, skills, rules, check-up).

## Ролевые агенты

Промт устанавливает 12 специализированных ролей, активируемых фразой «действуй как [роль]»:

| Роль | Когда использовать |
|---|---|
| Аналитик | PRD, спецификации, анализ требований |
| Архитектор | Проектирование архитектуры модификаций |
| Разработчик | Написание и модификация кода |
| Ревьюер кода | Ревью на баги и соответствие стандартам |
| Ревьюер архитектуры | Оценка архитектурных решений |
| Исправитель ошибок | Быстрое исправление ошибок |
| Оптимизатор | Анализ и оптимизация производительности |
| Менеджер метаданных | Создание/изменение объектов метаданных |
| Планировщик | Планирование реализации фич |
| Рефакторинг | Удаление мёртвого кода, устранение дубликатов |
| Тестировщик | Деплой и тестирование через веб-клиент |
| Техписатель | Документация для пользователей и администраторов |

## Обновление инструментария

В созданном проекте есть скрипты для обновления из upstream:

```bash
bash scripts/update-skills.sh         # обновить навыки cc-1c-skills
bash scripts/update-cursor-rules.sh   # обновить правила cursor_rules_1c
```
